# Video2 异步视频与创作中心模型能力设计

## 背景

中转站已经能够把 `video-2.0` 和 `video-2.0-fast` 路由到
`POST /v1/video/async-generations`，也能通过
`GET /v1/video/async-generations/{task_id}` 轮询任务。但是当前创作中心仍对大多数视频模型发送通用的 `seconds + size` 请求，并展示 1080、2K、4K 与 5/10/15 秒选项，不符合 [Video2 下游调用文档](https://docs.qq.com/doc/DTmNnT1BDa0FSWVJ1)。

本设计补齐 Video2 下游参数、校验和创作中心模型定制能力，同时保持现有 Sora2 与其他视频模型行为不变。

## 目标

- 支持 `video-2.0` 与 `video-2.0-fast` 的异步提交和轮询结果。
- 下游 JSON 请求可使用文档中的基础参数和远程参考素材参数。
- 创作中心根据所选模型展示正确的时长、画幅、清晰度和参考素材控件。
- 将每个模型的能力集中定义，避免在组件和 API 层持续堆叠模型名称条件。
- 对可在网关可靠判断的参数做一致校验，并保持请求原值透传。
- 不回归 Sora2 和其他现有视频模型的请求格式。

## 非目标

- 本次不建设对象存储或本地文件上传服务。
- 本次不下载远程素材来验证文件体积、视频时长或视频分辨率。
- 本次不把模型能力配置迁移到数据库或管理端。
- 本次不改变现有异步任务持久化、计费或轮询架构。

## 文档约束

两个 Video2 模型共有以下基础能力：

| 参数 | 约束 |
| --- | --- |
| `prompt` | 必填，去除首尾空白后非空，最长 5000 字符 |
| `duration` | 可选整数，4 至 15 秒 |
| `aspect_ratio` | 可选，`9:16`、`16:9`、`1:1` |
| `resolution` | 可选，仅 `720p` |
| `size` | 可选，仅 `720x1280`、`1280x720`、`720x720` |
| `async` | 可选布尔值；创作中心固定发送 `true` |
| 图片参考 | 总数最多 4 个，每个文件最大 20 MB，由上游验证体积 |
| 视频参考 | 总数最多 3 个，总大小最大 200 MB，总时长不超过 15 秒，由上游验证内容约束 |
| 音频参考 | 单个音频，2 至 15 秒，最大 15 MB，由上游验证内容约束 |

`size` 与 `aspect_ratio + resolution` 是两种等价的输出规格表达。创作中心使用后者，直接下游调用可任选其一；若同时提供，则两者必须表达相同方向和规格。

## 方案选择

采用“前端模型能力注册表 + 后端 Video2 专用校验”方案。

未采用直接模型条件分支，因为它会继续扩大 `session.ts` 和 `api.ts` 的职责。未采用后端动态下发能力，因为本次只有两个固定模型，扩展目录协议、配置存储和管理界面的成本明显超过收益。

## 模型能力矩阵

前端以规范化后的小写模型 ID 查找能力。没有注册的模型继续使用当前通用配置。

| 能力 | `video-2.0` | `video-2.0-fast` |
| --- | --- | --- |
| 时长 | 4 至 15，每秒可选 | 4 至 15，每秒可选 |
| 画幅 | `9:16`、`16:9`、`1:1` | `9:16`、`16:9`、`1:1` |
| 清晰度 | 仅 `720p` | 仅 `720p` |
| 单图或多图参考 | 开放 | 开放 |
| 首尾帧 | 不在创作中心开放 | 开放 |
| 单视频或多视频参考 | 不在创作中心开放 | 开放 |
| 音频参考 | 不在创作中心开放 | 开放 |
| 下游 API 高级字段透传 | 支持 | 支持 |

普通 `video-2.0` 的创作中心只开放两个模型文档中明确共有的图片参考能力。`video-2.0-fast` 开放文档示例覆盖的完整高级参考素材。直接调用下游 API 时，两个模型都允许这些字段通过网关校验与透传，最终能力由上游决定。

## 前端设计

### 能力与状态

在创作中心会话模块中新增视频能力描述与 Video2 专用选项：

- 时长使用数值状态，Video2 限制为 4 至 15。
- 画幅状态支持 `9:16`、`16:9`、`1:1`。
- 清晰度状态包含 `720p`，同时保留现有通用模型值以兼容历史记录。
- 参考素材状态使用明确命名的远程 URL 字段和 URL 列表。
- 模型切换时运行归一化函数；不支持的旧值重置为新模型默认值，不把隐藏字段带入请求。

默认值为 `duration: 4`、`aspectRatio: "9:16"`、`resolution: "720p"`。历史记录继续保存展示所需的时长和清晰度，并可选保存画幅；不持久化远程参考 URL，避免在浏览器历史中长期保存可能敏感的资源地址。

### 控件

选择 Video2 模型后，创作中心显示：

- 画幅：三个稳定尺寸的分段选项。
- 清晰度：唯一的 `720p` 选项，保持可见以明确实际输出规格。
- 时长：4 至 15 的数值步进控件。
- 参考素材：折叠区中的 URL 输入和可增删列表。

`video-2.0-fast` 的折叠区包含图片、首帧、尾帧、视频参考和音频。`video-2.0` 只显示图片参考。空字段不发送，模型切换后不支持的参考字段会被清理。

现有“添加素材”仍是浏览器本地预览。它生成的 `data:` URL 不会自动写入 Video2 远程参考参数，界面文案继续明确本地素材尚未接入上传服务。

### 请求构造

请求构造由纯函数负责，并返回最终提交负载与预计耗时：

- 通用视频模型继续发送 `{ model, prompt, seconds, size }`。
- Sora2 继续使用现有 4/8/12 秒与固定尺寸逻辑。
- Video2 发送 `{ model, prompt, duration, aspect_ratio, resolution, async: true }`。
- 远程参考参数只有在能力允许且值非空时加入 Video2 请求。

创作中心不同时发送 `size` 与 `aspect_ratio + resolution`，避免重复规格不一致。预计耗时沿用现有本地估算机制，以所选时长为主要输入，不影响计费。

## 后端设计

### 请求数据结构

为 Video2 校验定义专用请求结构，不把所有高级字段塞入通用任务 DTO。新增可选标量必须使用指针和 `omitempty`：

- `Duration *int`
- `Async *bool`
- 字符串、数组与对象字段按 JSON 原名定义。
- `video_reference` 使用元素结构 `{ "url": "..." }`。

解析与重编码继续使用 `common` JSON 包装函数。验证只读取结构化值；`BuildRequestBody` 仍以原始 JSON map 为基础，仅替换上游模型名，因此显式 `false` 和 `0` 不会被静默删除。

### 校验

当规范化模型名为 `video-2.0` 或 `video-2.0-fast` 时执行专用校验：

1. `prompt` 非空且不超过 5000 字符。
2. 已提供的 `duration` 必须是 4 至 15 的整数。
3. `aspect_ratio`、`resolution` 和 `size` 必须在白名单中。
4. 同时提供 `size` 与 `aspect_ratio` 时，方向必须一致；`resolution` 若存在必须为 `720p`。
5. `image_urls` 最多 4 个；将 `image_url`、`image_urls`、首帧和尾帧合并计算图片总数时也不得超过 4。
6. `video_reference` 最多 3 个；若同时提供 `video_url`，视频总数也不得超过 3。
7. 所有参考 URL 必须是绝对 `http` 或 `https` URL。

校验失败返回 HTTP 400 与 `invalid_request`。网关不发起远程 HEAD/GET 请求；文件大小、媒体时长、编码和分辨率错误由上游返回。

JSON 请求继续完整透传。Multipart 不是该 Video2 文档的目标请求格式，不为高级数组或对象参数新增 multipart 编码约定。

### 计费与轮询

现有预估计费读取 `duration` 的逻辑保持有效，默认时长继续为 4 秒。Video2 的 720p 三种尺寸按当前基础尺寸倍率处理；本次不改变固定费用或表达式计费配置。

提交仍进入 `/v1/video/async-generations`，轮询仍进入 `/v1/video/async-generations/{task_id}`。完成响应优先读取顶层 `url`，并保留现有 `metadata.url` 等兼容路径。

## 错误处理

- 前端在提交前校验范围、数量和 URL 格式，并给出具体字段错误。
- 后端重复执行相同的可确定校验，作为直接 API 调用的最终保护。
- 上游媒体内容错误保持原有中转错误响应，不在网关中猜测或改写。
- 模型能力表查找失败时回退到现有通用视频选项，避免未知模型无法使用。

## 国际化

新增界面文字使用英文源键并同步到 `en`、`zh`、`fr`、`ru`、`ja`、`vi` 六种语言。执行 `bun run i18n:sync` 后检查没有缺失键或直接硬编码的可见文案。

## 测试策略

实现采用测试驱动开发。

### 后端

- 两个 Video2 模型的完整合法请求通过校验并保持高级字段。
- `async: false` 与非法 `duration: 0` 都能被准确识别，证明显式零值没有丢失。
- 覆盖时长、画幅、清晰度、尺寸与规格冲突。
- 覆盖图片和视频总数上限、非 HTTP URL、空 prompt 和超长 prompt。
- 覆盖请求体最终仍使用映射后的上游模型名。
- 覆盖 Sora2 和未知异步视频模型不进入 Video2 专用校验。

### 前端

- 能力查找和模型切换归一化。
- Video2、Video2 Fast、Sora2 与通用模型的不同请求负载。
- Video2 的默认值、4 至 15 时长边界和三个画幅。
- 高级参考字段按模型能力加入或清理。
- URL 格式和图片、视频数量限制。

### 端到端验证

- 运行 Go 定向测试和受影响包测试。
- 运行前端测试、类型检查、i18n 同步检查和生产构建。
- 启动本地开发服务器，在桌面与移动视口分别切换两个 Video2 模型。
- 检查选项正确、文本不溢出、控件不重排、请求负载符合设计且浏览器控制台无错误。
- 使用可控测试上游或请求拦截验证提交后轮询并展示最终视频 URL。

## 完成标准

- 直接下游调用可提交 Video2 文档内的可选参数，并收到明确的本地校验错误或正常上游响应。
- 创作中心对两个 Video2 模型只展示 720p，不再展示 1080、2K 或 4K。
- 创作中心允许 4 至 15 秒和三个画幅，并发送 `duration + aspect_ratio + resolution + async`。
- `video-2.0-fast` 可提交完整远程参考素材参数，`video-2.0` 的界面只开放图片参考。
- 异步任务能够轮询到完成状态并展示最终视频 URL。
- Sora2 与其他现有视频模型的选项和请求格式不变。
- 后端测试、前端测试、类型检查、构建、i18n 与浏览器验证全部通过。
